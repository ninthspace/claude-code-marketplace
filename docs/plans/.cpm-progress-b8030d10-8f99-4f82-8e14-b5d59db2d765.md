## Discussion Highlights

### Key points so far

- Board launch names each tmux session `cpm-<project>-<suffix>`, where suffix = whole-second unix timestamp (`_default_session_suffix` in board.py:80).
- **Normal case (launches >1s apart):** distinct timestamp → distinct session name → two independent, coexisting tmux sessions. Both tracked in `_live_sessions` (keyed by name), both get a `● live` pill. This is by design — `action_launch` docstring: "good for running several sessions at once."
- Attach (`t`) targets the **newest** live session for the project (`_live_session_for`, board.py:822 — last insertion-order match).
- **Edge case (two launches within the same whole second):** identical suffix → identical session name → `tmux new-session -d -s <name>` on the second launch exits non-zero ("duplicate session"). But `_runner` is `subprocess.run` with no `check=True` and the return code is never inspected (board.py:1100) → failure is **silent**. Code proceeds: `_live_sessions[name]` overwrite is a no-op; detached mode then `tmux attach -t =<name>` drops you into the **first/existing** session, not a new one. Toast still says "Launched…". You believe you started a fresh session but you're in the old one.
